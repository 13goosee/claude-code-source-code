# 從 Claude Code 原始碼看可擴展的一人架構

> 基於 Claude Code v2.1.88 原始碼，解析「一個人怎麼做出能不斷長大的產品」
> 寫給想做一人公司的 builder，不綁定任何特定產品

---

## 為什麼這篇值得讀

一人公司最大的限制不是錢，是手。你只有一雙手，但產品要長大，功能要增加，使用者要的東西越來越多。

傳統解法是請人。但如果你不想請人呢？

Anthropic 當然不是一人公司。但他們在 Claude Code 裡做了一件很聰明的事：**把產品設計成讓別人幫你擴展的結構。** 核心團隊只維護骨架，具體的能力來自外部——社群寫的、使用者自己裝的、第三方接的。

這篇文章拆解的是：他們怎麼設計這些「讓別人幫你長大」的機制？這些設計的哲學，對只有一個人的你，有什麼用？

程式碼不會說謊。我們直接從原始碼裡看。

---

## 一、MCP——讓全世界幫你做功能的萬用插座

### 這是什麼？

MCP 是 Model Context Protocol 的縮寫。用白話講：**它是一個萬用插座，讓任何外部工具都能接進來。**

你家裡的電器不需要知道電廠怎麼發電，只要插頭符合規格就能用。MCP 做的就是定義這個「插頭規格」——只要你的工具符合協議，就能被 AI 使用。

### 原始碼怎麼做的？

打開 `src/services/mcp/types.ts`，第一個映入眼簾的是傳輸方式的定義：

```typescript
export type Transport = 'stdio' | 'sse' | 'sse-ide' | 'http' | 'ws' | 'sdk'
```

六種接法。本地程式可以用 `stdio`（直接對話），遠端服務可以用 `http`（網路請求），甚至同一個程式內部都可以用 `sdk`（進程內通訊）。

這代表什麼？**不管你的工具是一個 Python 腳本、一個雲端 API、還是一個跑在瀏覽器裡的擴充功能，都能接上來。**

再看設定的分層。`src/services/mcp/config.ts` 定義了七個設定來源，從低優先到高：

| 優先順序 | 來源 | 誰設定的 |
|---------|------|---------|
| 最低 | Claude.ai 連接器 | 平台方（Anthropic） |
| ↓ | Plugin 動態提供 | 生態系夥伴 |
| ↓ | 使用者全域設定 | 使用者自己 |
| ↓ | 專案設定 | 專案層級 |
| ↓ | .mcp.json | 專案根目錄 |
| ↑ | 企業管控 | IT 管理員 |
| 最高 | 託管設定 | 組織政策 |

這個分層的巧妙之處：**每一層都可以獨立運作，互不干擾。** 使用者可以裝自己的工具，企業可以覆蓋不安全的工具，專案可以有專案特定的設定。沒有人需要改原始碼。

### 工具怎麼被發現的？

`src/services/mcp/client.ts` 裡的 `getMcpToolsCommandsAndResources()` 函式做了一件很聰明的事——它把本地工具和遠端工具分開處理：

- **本地工具**（需要啟動程式）：一次最多同時處理 3 個，因為開程式有成本
- **遠端工具**（只是網路請求）：一次同時處理 20 個，因為只是等回應

對每個接上來的工具，系統會：
1. 檢查有沒有被政策禁止
2. 檢查需不需要授權（OAuth）
3. 連線
4. 拉回工具清單
5. 把外部工具轉成內部格式

轉換的關鍵在命名。`src/services/mcp/mcpStringUtils.ts` 裡，每個外部工具都會被加上前綴：

```typescript
buildMcpToolName("slack", "send_message") → "mcp__slack__send_message"
```

**外部工具進到系統後，看起來跟內建工具一模一樣。** AI 不需要知道這個工具是自家寫的還是別人接的。

### 對一人公司的啟示

你不需要自己做所有功能。你需要的是設計一個好的插座。

MCP 的設計教我們三件事：
1. **定義標準，不定義實作。** 你定義「什麼東西可以插進來」的規格，具體怎麼做留給別人。
2. **分層設定讓不同角色各取所需。** 你的核心使用者、企業客戶、生態系夥伴，可以在不衝突的前提下各自擴展。
3. **外部工具進來後要變成「自己人」。** 使用者不該感覺到哪個功能是原生的、哪個是外掛的。體驗要一致。

## 二、Skills——把智慧包裝成食譜，讓任何人都能貢獻

### 這是什麼？

如果 MCP 是讓工具接進來的插座，Skills 就是讓「做事的方法」可以被分享的食譜書。

一個 Skill 是一段 Prompt 加上流程，包裝成可重複使用的技能包。你可以想像成：**把一個老手的工作方式寫成 SOP，新手按照 SOP 就能做到八成功力。**

### 原始碼怎麼做的？

`src/skills/bundledSkills.ts` 定義了 Skill 的資料結構：

```typescript
type BundledSkillDefinition = {
  name: string                    // 技能名稱
  description: string             // 一行描述
  whenToUse?: string              // 什麼時候用
  allowedTools?: string[]         // 可以用哪些工具
  context?: 'inline' | 'fork'    // 在原對話執行，還是開新的
  getPromptForCommand: (args, context) => Promise<ContentBlockParam[]>
}
```

注意 `context` 這個欄位。`inline` 代表直接在當前對話裡執行（像是你在跟同事說「照這個方式做」），`fork` 代表開一個新的子對話去執行（像是你把任務丟給另一個人去做）。

**一個 Skill 可以很簡單——就是一段 Markdown 文字。** 放在 `.claude/skills/my-skill/SKILL.md`，系統就會自動發現它。

`src/skills/loadSkillsDir.ts` 裡的發現機制更有意思。Skills 從四個地方載入：

1. **託管目錄**（組織統一管理）
2. **使用者目錄**（`~/.claude/skills/`，你個人的技能庫）
3. **專案目錄**（`.claude/skills/`，專案共用的技能）
4. **動態目錄**（命令列指定的額外路徑）

重複的技能？第一個找到的贏。託管 > 使用者 > 專案。

### 最聰明的設計：條件觸發

Skills 支援 `paths` 設定，用 gitignore 風格的路徑匹配：

```yaml
---
name: Python Linter
when_to_use: When linting Python files
paths: "**/*.py"
---
```

這個 Skill 平常不會出現在 AI 的工具列表裡。**只有當你碰到 `.py` 檔案時，它才會自動啟用。**

為什麼這很重要？因為 AI 的注意力是有限的（context window）。如果你一次塞 100 個 Skill 進去，AI 反而不知道該用哪個。條件觸發讓 Skill 在「對的時機」出現，而不是一直佔位子。

### Skill 列表的預算控制

`src/tools/SkillTool/prompt.ts` 裡有一個精妙的設計——Skill 的描述只佔 context window 的 1%：

- 先嘗試顯示完整描述
- 超過預算就按比例截短
- 再超就只列名字
- 內建 Skill 的描述永遠不截短（它們是核心功能）

**這是一個「有限資源下如何分配注意力」的經典設計。** 跟一人公司的資源分配邏輯一模一樣——核心的東西要保護，非核心的按重要性排序。

### 對一人公司的啟示

Skills 教我們：**你的知識和方法論可以被包裝成資產。**

1. **讓使用者自己定義「怎麼用你的產品」。** 不要把所有使用方式都寫死。給使用者一個框架，讓他們寫自己的 SOP。
2. **讓社群貢獻智慧，而不只是回報 bug。** 一個好的 Skill 是使用者的實戰經驗濃縮，這比你自己想像的 use case 有價值一百倍。
3. **條件觸發 > 全部塞進去。** 在對的時機提供對的東西，比什麼都提供更有用。

## 三、Hooks——讓使用者不改你的程式碼就能改你的行為

### 這是什麼？

想像你買了一台車。Skills 是讓你加裝行車記錄器、手機架這些配件。Hooks 不一樣——**Hooks 是讓你改變車子本身的行為。**

踩煞車之前，先檢查有沒有繫安全帶。啟動引擎之後，自動播放你喜歡的音樂。你沒有改裝引擎，但車子的行為變了。

### 原始碼怎麼做的？

`src/utils/hooks/hooksConfigManager.ts` 定義了 25 種生命週期事件。不是 2、3 種，是 25 種：

| 事件 | 時機 | 可以做什麼 |
|------|------|-----------|
| `PreToolUse` | 工具執行前 | 攔截、修改輸入、阻止執行 |
| `PostToolUse` | 工具執行後 | 檢查結果、追加處理 |
| `UserPromptSubmit` | 使用者送出提問時 | 過濾敏感資訊、注入額外 context |
| `SessionStart` | 對話開始 | 初始化環境、載入設定 |
| `SubagentStart` | 子 Agent 啟動 | 監控、限制子任務 |
| `ConfigChange` | 設定檔變更 | 即時回應設定更新 |
| `FileChanged` | 檔案被修改 | 自動 lint、自動測試 |
| ... | ... | ... |

25 個事件覆蓋了從「使用者按下 Enter」到「AI 回覆結束」的每一個節點。

更猛的是 Hook 的執行方式。不只是 shell command，支援五種：

| 類型 | 怎麼執行 | 適合什麼 |
|------|---------|---------|
| `command` | 跑 shell 指令 | 簡單檢查、呼叫腳本 |
| `prompt` | 用另一個 AI 判斷 | 需要理解語意的檢查 |
| `agent` | 啟動一個完整 Agent | 複雜的自動化流程 |
| `http` | 發 HTTP 請求 | 跟外部服務互動 |
| `function` | TypeScript callback | 程式內部的邏輯擴展 |

**用 AI 來判斷 AI 的行為——這是 Hook 最前衛的設計。** `prompt` 類型的 Hook 會啟動一個小型 AI（Haiku 模型）來決定「要不要讓主 AI 執行這個動作」。

### Hook 的輸出協議

Hook 不只是「通過/不通過」，它可以回傳結構化的 JSON：

```json
{
  "decision": "approve" 或 "block",
  "reason": "為什麼",
  "hookSpecificOutput": {
    "updatedInput": { "修改過的輸入" },
    "permissionDecision": "allow" 或 "deny" 或 "ask"
  }
}
```

一個 Hook 可以：
- **阻止**動作（`block`）
- **修改**動作的輸入（`updatedInput`）
- **覆蓋**權限決策（`permissionDecision`）
- **注入**額外的系統訊息（`systemMessage`）

但有一條安全底線：**Hook 的 `allow` 不能覆蓋設定檔裡的 `deny`。** 你可以用 Hook 放行，但你不能用 Hook 繞過明確禁止的規則。

### 對一人公司的啟示

Hooks 教我們的是**可程式化的行為邊界**。

1. **不要只設計功能，要設計「可以被插入邏輯的接縫」。** 你不知道使用者會怎麼用你的產品，但你可以在關鍵節點留下接口。
2. **讓使用者控制行為而不碰原始碼。** 改設定檔比改程式碼門檻低一百倍。大部分使用者不會寫程式，但會寫設定。
3. **安全邊界要硬編碼。** 給使用者自由，但留一條不能跨越的線。自由和安全不矛盾，關鍵在設計。

## 四、AgentTool 套娃——AI 叫另一個 AI 去做子任務

### 這是什麼？

你是一個老闆。你不可能每件事都自己做，你會找人去做。但你不會把所有權限都給他——你會說清楚能做什麼、不能做什麼，做完回來跟你報告。

AgentTool 就是這件事的 AI 版本。**一個 AI 可以啟動另一個 AI 來做子任務。**

### 原始碼怎麼做的？

`src/tools/AgentTool/AgentTool.tsx` 定義了啟動子 Agent 的輸入：

```typescript
const baseInputSchema = z.object({
  description: z.string(),      // 3-5 字摘要
  prompt: z.string(),           // 完整的任務描述
  subagent_type: z.string(),    // 什麼類型的專家
  model: z.enum(['sonnet', 'opus', 'haiku']),  // 用哪個模型
  run_in_background: z.boolean()  // 要不要背景執行
})
```

注意 `subagent_type`。系統內建了多種專家：

| 類型 | 專長 | 工具限制 |
|------|------|---------|
| `general-purpose` | 萬用 | 幾乎全部 |
| `Explore` | 搜尋和研究 | 只讀，不能編輯 |
| `Plan` | 設計和規劃 | 只讀，不能編輯 |
| `claude-code-guide` | 使用說明 | 只讀，不能編輯 |

**不同類型的子 Agent 有不同的工具權限。** 一個只負責「看」的 Agent 不能「改」東西。這是最小權限原則的完美體現。

### Fork 機制——共享前文，各做各的

`src/tools/AgentTool/forkSubagent.ts` 裡有一個精妙的 Fork 設計：

子 Agent 會繼承父 Agent 的完整對話記錄，但從 fork 點開始分道揚鑣。就像 Git 的 branch——分支前的歷史共享，分支後各走各路。

更聰明的是快取利用。因為分支前的對話完全一樣，API 的 prompt cache 可以共用。**你啟動 5 個子 Agent，前面共用的部分只需要計算一次。** 省錢。

### 子 Agent 不能做什麼？

`src/tools/AgentTool/agentToolUtils.ts` 裡定義了禁止清單：

- **子 Agent 不能再啟動子 Agent**（防止無限套娃）
- **背景 Agent 只能用特定工具**（防止失控）
- **自訂 Agent 比內建 Agent 的權限更少**（信任分級）

這裡的設計哲學是：**自由度和信任成正比。** 你自己信得過的（內建的）可以做更多事，外面來的（社群貢獻的）限制更嚴。

### 對一人公司的啟示

AgentTool 教我們**委派的藝術**。

1. **定義角色，不定義步驟。** 你告訴子 Agent「你是一個研究員」，而不是「第一步搜尋、第二步過濾、第三步整理」。讓 AI 自己決定怎麼做。
2. **權限跟著角色走。** 研究員可以看不能改，執行者可以改但要問。不要給所有人所有權限。
3. **背景執行是生產力倍增器。** 一個人做不了的事，可以同時派五個 AI 去做。但背景任務要有更嚴格的限制——因為你不在旁邊看。

## 五、40+ 個工具的管理——註冊、發現、和延遲載入

### 這是什麼？

Claude Code 有超過 70 個工具。加上使用者透過 MCP 接進來的，可能上百個。你不可能把 100 個工具的說明書全部塞進 AI 的腦袋裡——**它會迷路的。**

這就像你走進一間什麼都賣的百貨公司。如果一進門就看到所有商品，你反而找不到要買的東西。好的百貨公司會分樓層、分區域、有導覽圖。

### 原始碼怎麼做的？

`src/Tool.ts` 定義了每個工具的關鍵屬性：

```typescript
// 工具的核心屬性
name: string              // 唯一名稱
searchHint?: string       // 3-10 字的能力描述（給搜尋用的）
shouldDefer?: boolean     // 要不要延遲載入
alwaysLoad?: boolean      // 永遠載入（不延遲）
```

`shouldDefer` 這個布林值是整個系統的關鍵開關。設為 `true` 的工具不會馬上載入完整定義——AI 只知道它的名字，不知道怎麼用。**需要的時候再查。**

`src/tools.ts` 是工具的中央登記處。它用了幾個聰明的技巧：

**條件載入：**
```typescript
const SleepTool = feature('KAIROS')
  ? require('./tools/SleepTool/SleepTool.js').SleepTool
  : null

const cronTools = feature('AGENT_TRIGGERS')
  ? [CronCreateTool, CronDeleteTool, CronListTool]
  : []
```

功能沒開的工具根本不會被載入。不是「載入了但不能用」，是**完全不存在**。這讓核心保持精簡。

**延遲載入打破循環依賴：**
```typescript
// 不能直接 import，因為會造成 A 依賴 B、B 依賴 A 的迴圈
const getTeamCreateTool = () =>
  require('./tools/TeamCreateTool/TeamCreateTool.js').TeamCreateTool
```

這是一個很實際的工程問題：當工具太多，互相之間的依賴會打結。解法是「需要的時候再取」，而不是「啟動時全部準備好」。

### ToolSearch——AI 界的 Google

`src/tools/ToolSearchTool/ToolSearchTool.ts` 是整個延遲載入的核心。當 AI 需要一個被延遲的工具時，它用 ToolSearch 來找：

兩種查法：
- **精確查找**：`select:WebSearch,WebFetch`——直接指名
- **關鍵字搜尋**：`notebook jupyter`——按能力搜尋

搜尋的計分系統：

| 匹配位置 | 分數 |
|---------|------|
| 工具名稱完全匹配 | +10 分 |
| 工具名稱部分匹配 | +5 分 |
| searchHint 匹配 | +4 分 |
| 描述關鍵字匹配 | +2 分 |
| MCP 工具額外加分 | +2 分 |

**為什麼 MCP 工具有額外加分？** 因為 MCP 工具全部都是延遲載入的，如果不加分，它們會被內建工具壓過去。這是一個「補償弱勢」的設計——確保外部貢獻的工具有被發現的機會。

### 自動延遲的閾值

系統甚至有自動判斷要不要啟用延遲載入的機制。`src/utils/toolSearch.ts` 計算延遲工具佔 context window 的比例：

- **超過 10%**：自動啟用 ToolSearch，開始延遲載入
- **低於 10%**：全部工具直接載入，不需要搜尋

**系統會根據「腦袋裝不裝得下」來自動決定要不要分批載入。** 工具少就全塞，工具多就按需載入。

### 對一人公司的啟示

工具管理教我們**有限注意力下的資訊架構**。

1. **不是所有東西都要一次展示。** 你的產品功能越多，使用者越需要被引導。按需呈現比全部列出更有用。
2. **讓好東西能被找到。** 再好的功能，埋在選單第五層就沒人用。設計搜尋機制，讓功能主動浮現。
3. **外部貢獻的東西需要被「推一把」。** 如果你建了一個平台，第三方貢獻的內容天然處於劣勢（使用者不知道它存在）。你需要在系統層面幫它一把。

## 六、Coordinator Mode——一個人指揮一群 AI

### 這是什麼？

前面的 AgentTool 是「一個 AI 叫一個 AI 做事」。Coordinator Mode 更極端——**一個 AI 變成指揮官，管理一群 AI 同時做事。**

你是交響樂團的指揮。你不演奏任何樂器，但你決定誰什麼時候演奏什麼。

### 原始碼怎麼做的？

`src/coordinator/coordinatorMode.ts` 定義了指揮官可以用的工具：

```typescript
// 指揮官只有四把刷子
COORDINATOR_MODE_ALLOWED_TOOLS = {
  AgentTool,           // 派人去做
  TaskStopTool,        // 叫停
  SendMessageTool,     // 跟進
  SyntheticOutputTool  // 整合結果
}
```

**注意它不能用的：** 指揮官不能自己讀檔案、不能自己寫程式碼、不能自己搜尋。它只能透過工人 Agent 來做這些事。

這是一個非常刻意的設計限制：**指揮官的工作是思考和協調，不是動手。** 如果指揮官自己開始改程式碼，它就不再是指揮官了，它變成了一個什麼都做但什麼都做不好的全能工人。

工人 Agent 也有限制。它們可以讀寫檔案、搜尋、執行指令，但**不能再啟動子 Agent**（防止無限套娃）。

### 任務完成的通知格式

工人做完後，用結構化格式回報：

```xml
<task-notification>
  <task-id>agent-123</task-id>
  <status>completed</status>
  <summary>已完成 API 重構</summary>
  <result>具體結果...</result>
  <usage>
    <total_tokens>15000</total_tokens>
    <tool_uses>8</tool_uses>
    <duration_ms>45000</duration_ms>
  </usage>
</task-notification>
```

系統 prompt 裡有一條關鍵指令：

> 「你必須閱讀工人的發現、理解它們，然後才寫出具體的規格。永遠不要把理解的工作委派出去。」

**指揮官可以委派執行，但不能委派理解。** 這是 Coordinator Mode 最深刻的設計原則。

### 對一人公司的啟示

Coordinator Mode 教我們**一個人如何管理多個 AI 的藝術**。

1. **管理者不動手。** 一旦你開始動手做具體的事，你就沒有精力去協調全局。AI 時代的一人公司，你的角色是指揮官，不是工人。
2. **理解不能外包。** 你可以讓 AI 幫你做研究、寫報告、跑數據。但最終的判斷和理解，必須是你的。這是一人公司老闆唯一不能委派的工作。
3. **結構化的回報機制。** 當你同時讓多個 AI 做事，你需要一致的回報格式——否則你會迷失在混亂的資訊裡。

## 七、共同的設計哲學：核心小、介面穩定、擴展交給外部

把這七套機制攤開看，有一個非常一致的設計哲學：

### 原則一：核心只做路由，不做業務

Claude Code 的核心不「知道」怎麼搜尋網頁、不「知道」怎麼讀 Jupyter notebook、不「知道」怎麼跟 Slack 互動。核心只知道一件事：**怎麼把正確的任務送到正確的工具。**

```
使用者的意圖 → 核心路由 → 工具執行 → 結果回傳
```

所有具體的能力都在「工具」這一層，而工具可以來自任何地方。

### 原則二：介面穩定，實作自由

每個擴展機制都有清晰的「契約」：

| 機制 | 契約 |
|------|------|
| MCP | 標準協議，任何語言都能實作 |
| Skills | Markdown 檔案 + Frontmatter |
| Hooks | JSON 設定 + 結構化輸出 |
| Tools | `buildTool()` 工廠函式 |
| Agents | 目錄結構 + 定義檔 |

**只要你遵守契約，用什麼語言寫、跑在哪裡、怎麼實作，系統都不管。**

### 原則三：不同信任等級，不同自由度

| 來源 | 信任等級 | 自由度 |
|------|---------|--------|
| 核心內建 | 最高 | 全部權限 |
| 使用者自定 | 中高 | 大部分權限 |
| 專案設定 | 中 | 需要工作區信任 |
| 社群/第三方 | 低 | 受限權限 |
| 託管/企業 | 特殊 | 可以覆蓋一切 |

這不是歧視，是務實。**你自己寫的東西你自己負責，別人寫的東西要多一層把關。**

### 原則四：延遲做決定

- ToolSearch 延遲載入工具定義
- Skills 延遲啟用條件觸發的技能
- MCP 延遲連線遠端服務
- `lazySchema()` 延遲解析輸入格式
- `require()` 延遲載入模組

到處都是「不急的事先不做」。**這不是懶，這是聰明。** 你不知道使用者會用到什麼，先準備一切是浪費資源。

## 八、對一人公司的完整啟示

### 什麼該自己做

從原始碼的設計決策裡，可以看出 Anthropic 自己做了什麼：

- **協議設計**（MCP 的規格）
- **路由邏輯**（哪個工具處理哪個任務）
- **信任機制**（什麼可以做、什麼不能做）
- **資源管理**（context window 的分配、快取的利用）

這些都是「基礎設施」——看不見的東西，但決定了整個系統的品質上限。

### 什麼該讓別人做

- **具體的工具實作**（透過 MCP 接入）
- **使用方法和流程**（透過 Skills 分享）
- **行為自定義**（透過 Hooks 設定）
- **重複性的執行工作**（透過 Agent 委派）

### 怎麼設計「插孔」

1. **先想「誰會來擴展」。** 使用者？開發者？企業 IT？不同的人需要不同的介面複雜度。Skills 用 Markdown（人人都會），MCP 用 JSON 協議（開發者會），Hooks 用設定檔（進階使用者會）。

2. **先想「出事了怎麼辦」。** 每個擴展機制都有安全邊界。MCP 有政策過濾、Skills 有權限限制、Hooks 有信任檢查、Agent 有工具白名單。**在開放之前先想好怎麼關門。**

3. **先想「怎麼被發現」。** 再好的擴展，沒人知道就沒人用。ToolSearch 的計分系統、Skills 的條件觸發、Agent 的類型列表——都是讓正確的東西在正確的時機被看到的機制。

### 平台思維 vs 產品思維

Claude Code 的原始碼揭示了一個漸進策略：

**先做產品（解決具體問題），再長出平台（讓別人也能在上面解決問題）。**

它一開始就是一個 AI coding assistant——這是產品。但它的內部架構從一開始就預留了擴展點——MCP、Skills、Hooks、Agent 都不是後來硬塞的，是骨架本身就支撐的。

**一人公司的正確順序是：**

1. 先解決一個具體問題（產品）
2. 解決問題的方式設計成可複製的（模式）
3. 讓別人也能用你的方式解決類似問題（平台）

不要一開始就做平台。沒有人會在一個空的平台上建東西。先有價值，再開放能力。

## 一句話總結

Claude Code 的原始碼告訴我們：**一個人做不出什麼都能做的產品，但一個人可以做出一個讓全世界幫你擴展的骨架。**

核心要小到你一個人維護得動。介面要穩到別人敢在上面建東西。擴展要方便到不會寫程式的人也能參與。安全要硬到你敢放手讓別人進來。

這不是技術決策，這是商業決策。你在設計的不是程式碼架構，是**價值的生產方式**。
