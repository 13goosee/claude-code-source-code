# 從 Claude Code 原始碼看軟體架構哲學

> 基於 Claude Code v2.1.88 原始碼的真實設計決策
> 寫給不寫程式但想理解設計思維的人，不綁定任何特定產品

---

## 一個論點

多數人以為好的軟體架構是讓系統「完美運作」的技術。

不是。

看完 Anthropic 的 1,900 個原始碼檔案，我發現好的架構只做一件事：**讓系統在不完美的世界裡，優雅地運作。**

API 會掛、模型會退化、網路會斷、使用者會做你沒想到的事。這些不是意外，是常態。架構的工作不是阻止這些事發生，而是當它們發生時，系統還能站著。

這個信念在 Claude Code 裡化成四根柱子。每一根都不是理論——是原始碼裡可以直接讀到的設計決策。

---

## 支柱一：預期失敗

> 不是「如果」出錯，是「當」出錯的時候怎麼辦。

### 耐心是一種策略

當 API 呼叫失敗，Claude Code 不會立刻重試，也不會放棄。它用「指數退避」：

```
第一次失敗：等 0.5 秒再試
第二次失敗：等 1 秒
第三次：等 2 秒
第四次：等 4 秒
……最多等 32 秒
```

而且每次等待都加上 0-25% 的隨機擾動。為什麼？想像 1,000 個使用者同時遇到錯誤，如果每個人在完全相同的時間重試，就會一波又一波衝擊伺服器，讓問題更難恢復。加上隨機擾動，1,000 次重試會分散在一個時間窗裡。

這在經濟學裡叫**避免同步化行為**——央行不希望所有銀行在同一天到期同類資產，理由一樣。

### 面對錯誤，有時候正確的反應是「不要嘗試修復」

不是所有錯誤都值得重試。原始碼裡有一段刻意的分類：

```
// Foreground query sources where the user IS blocking on the result —
// these retry on 529. Everything else (summaries, titles, suggestions)
// bails immediately: during a capacity cascade each retry is 3-10×
// gateway amplification.
```

**使用者正在等的請求，值得重試。使用者沒在看的背景請求，應該立刻放棄。**

為什麼？伺服器過載時，每次重試都會讓問題惡化——原始碼直接寫了「3-10 倍閘道放大」。重試背景任務就像高速公路塞車時猛按喇叭——不但沒幫助，還讓全體更慢。

這是一個反直覺的工程判斷：**不行動，本身就是一種行動。**

### 偵測「假裝沒事」的故障

API 有時候不是「斷開」，而是「假裝還連著但不給資料」。這種故障最危險——因為系統不知道自己出了問題。

Claude Code 的解法是一個 90 秒的「看門狗計時器」。原始碼的註解解釋了為什麼：

```
// Streaming idle timeout watchdog: abort the stream if no chunks arrive
// for STREAM_IDLE_TIMEOUT_MS. Without this, a silently dropped connection
// can hang the session indefinitely since the SDK's request timeout only
// covers the initial fetch(), not the streaming body.
```

SDK 內建的超時機制只管「連線有沒有建立」，不管「連線建立後資料有沒有持續流入」。就像你打電話給客服，電話接通了但對方一直靜音——你的手機不會自動掛斷，因為「通話中」是正常狀態。所以你需要自己設一個鬧鐘。

而且這不是一刀切。45 秒先記一筆預警，90 秒才真正斷開。斷開之後也不是放棄——它會從串流模式降級到非串流模式，放棄即時顯示，改成等完整回覆再一次顯示。**不是「停止服務」，是「降低服務品質但保持服務」。**

### 主力不行就換替補

當 API 連續 3 次回傳「伺服器過載」（HTTP 529），系統自動切換到替補模型：

- Opus 4.6 → Opus 4.1
- Sonnet 4.6 → Sonnet 4.5
- Sonnet 4.5 → Sonnet 4.0

使用者可能會發現 AI 沒那麼聰明了，但至少還能用。這在商業上叫**服務等級降級**——航空公司超賣時先保證大家都能飛，再想辦法升等。

### 為「真實世界」設計，不是為「理想實驗室」

`errorUtils.ts` 裡列了 29 種 SSL/TLS 錯誤碼——`CERT_HAS_EXPIRED`、`SELF_SIGNED_CERT_IN_CHAIN`、`HOSTNAME_MISMATCH`——全都是企業用戶在公司代理伺服器後面會遇到的問題。而且錯誤處理邏輯會沿著錯誤的因果鏈往下追蹤，最多追 5 層，找到根本原因。

這意味著 Anthropic 的工程師預期使用者在充滿代理伺服器、防火牆、過期憑證的真實企業環境中使用——不是乾淨的開發環境。

**可遷移的思維：** 你的產品是為理想條件設計的，還是為真實世界設計的？如果你只在理想條件下測試，你的市場就只有理想條件。

### 預期失敗的整體畫面

把這些放在一起看，形成了一個完整的防禦體系：

| 故障類型 | 系統的反應 | 哲學 |
|---------|----------|------|
| API 暫時不可用 | 指數退避，越等越久 | 給對方恢復的時間 |
| 伺服器過載 | 前台重試，背景放棄 | 不要火上加油 |
| 連線假死 | 90 秒看門狗主動切斷 | 不信任「正常」的假象 |
| 模型持續過載 | 自動切換替補模型 | 有服務比完美更重要 |
| 網路環境惡劣 | 追蹤 5 層因果鏈 + 29 種錯誤碼 | 為真實世界設計 |

這不是悲觀主義，是務實主義。不是「一切都會壞」，而是「一切都有壞的可能，壞的時候我已經想好怎麼辦了」。

## 支柱二：隔離影響

> 火災不可怕，可怕的是整棟樓沒有防火牆。

預期失敗只是第一步。更重要的是：**當某個地方出問題時，影響不要擴散到其他地方。**

### 為什麼一個 22 行的檔案，是整個架構裡最精妙的設計

Claude Code 有一個叫 `autoDream` 的功能——受睡眠機制啟發的記憶整合引擎，在背景自動整理使用者的記憶。它的主程式有 325 行，會拉進一大堆重型依賴：子 agent 引擎、任務管理系統、訊息建構器……

但系統很多地方只需要問一個簡單的問題：**autoDream 有沒有開啟？**

如果每次問這個問題都要把整個 autoDream 引擎載進來，就像你只是想問「餐廳今天有開嗎」，結果服務生把整個廚房搬到你面前。

所以 Anthropic 把這個問題拆成一個獨立的 22 行小檔案——`config.ts`：

```
// Leaf config module — intentionally minimal imports so UI components
// can read the auto-dream enabled state without dragging in the forked
// agent / task registry / message builder chain that autoDream.ts pulls in.
```

**「這個檔案故意保持極簡，這樣 UI 元件只需要知道 autoDream 有沒有開，不用把整條依賴鏈都拖進來。」**

22 行的 config 只需要兩個輕量的資訊來源。325 行的主程式反過來從 config 讀取開關。兩邊互不干擾。

這就是隔離的核心：**知道答案的人，跟做事的人，不需要是同一個人。**

### 循環依賴：A 等 B、B 等 C、C 等 A

這個模式在整個原始碼裡反覆出現，因為大型系統最常見的結構性問題就是**循環依賴**——A 模組需要 B，B 需要 C，C 又需要 A，形成死結。

`main.tsx` 裡有一段刻意用「延遲載入」打斷這個循環：

```
// Lazy require to avoid circular dependency:
// teammate.ts -> AppState.tsx -> ... -> main.tsx
const getTeammateUtils = () => require('./utils/teammate.js')
```

另一個例子是 `cleanupRegistry.ts`——只有 26 行，開頭的註解直接說：

```
// Global registry for cleanup functions that should run during graceful shutdown.
// This module is separate from gracefulShutdown.ts to avoid circular dependencies.
```

**「這個模組跟 gracefulShutdown.ts 分開，就是為了避免循環依賴。」**

想像一家公司的組織圖。如果 A 部門的運作需要 B 部門，B 需要 C，C 又需要 A——什麼決策都做不了，因為每個部門都在等其他部門。

Anthropic 的解法是：**找出環上最輕的那個節點，把它獨立出來。** config 檔案、cleanupRegistry——都是那個「最輕的節點」，不依賴任何重型模組，所以任何人都可以安全地引用。

**在任何複雜系統裡——軟體、組織、供應鏈——當依賴形成環路時，解法不是消除依賴，而是在環上找到最小的斷點，把它抽出來。**

### 一個事實來源：消滅「版本不一致」的 bug

隔離不只是防止程式碼互相拖累。更深層的隔離是**防止資訊矛盾**。

Claude Code 整個應用程式只有一個「事實的來源」——一個集中的狀態存儲，包含使用者設定、對話歷史、可用工具、背景任務、通知佇列、協作 agent、UI 狀態。所有人都從同一個地方讀取，所有人都透過同一個機制更新。沒有人可以繞過。

為什麼這很重要？因為複雜系統裡最常見的 bug 不是「功能壞了」，而是**「兩個地方對同一件事有不同的理解」**——使用者以為自己在自動模式，但權限系統以為他在手動模式。一個模組說聊天記錄有 10 則，另一個說 11 則。

一個事實來源消除了這整類的 bug。

**商業類比：** 業務用 Salesforce、行銷用 HubSpot、客服用 Zendesk——三套系統裡的「客戶資料」會慢慢 drift 開來。一個客戶在 Salesforce 裡是「付費用戶」，在 Zendesk 裡還是「試用中」。唯一的解法是大家都用同一個系統。

## 支柱三：宣告而非控制

> 不要告訴別人怎麼做，讓他們告訴你他們是什麼。

前兩個支柱處理的是防禦——怎麼讓壞事不會變得更壞。這個支柱處理的是擴展——**怎麼讓系統越來越大，但不越來越難管。**

### 44 個工具，一份合約

Claude Code 有 44 個工具——讀檔案、跑終端指令、搜尋程式碼、瀏覽網頁。問題是：怎麼讓 44 個不同的工具保持一致的行為？

答案不是「寫一個很大的 if-else 來處理每個工具」。答案是**統一合約**。

每個工具都必須宣告自己：

- **我需要什麼輸入**（`inputSchema`）
- **我能跟別人同時跑嗎**（`isConcurrencySafe`）
- **我只是在看，還是會改東西**（`isReadOnly`）
- **我有沒有破壞力**（`isDestructive`）
- **誰可以叫我做事**（`checkPermissions`）

注意這裡的措辭：工具宣告的是「**我是什麼**」，不是「**你要怎麼對我**」。

系統的排程引擎看到這些宣告，自動決定執行策略。當 AI 說「我要同時搜尋三個檔案」，系統知道搜尋工具宣告了「唯讀」且「可並行」，三次搜尋可以同時跑。但如果 AI 說「先寫一個檔案，然後讀它」，系統知道寫入工具宣告了「有破壞力」，必須先完成才能繼續。

**「我是什麼」比「你要怎麼對我」更穩定。** 一個工具永遠是唯讀或可寫的——這個事實不會變。但系統怎麼對待唯讀工具可能會變（也許以後唯讀工具需要更快的超時設定）。把事實放在工具身上、把決策放在系統身上，當決策邏輯改變時，44 個工具不需要動。

**這跟好的團隊管理一樣：** 主管不應該需要知道每個下屬的工作細節才能分配任務。下屬應該主動告訴主管「這件事我可以跟別人平行做」或「這件事需要等某個結果出來」。工具合約就是每個工具交給系統的一張「能力卡片」。

### 想加新工具？不需要改核心

這個設計最大的好處是擴充成本極低。想加一個新工具：

1. 建一個新資料夾，寫一個符合合約的檔案
2. 在工具清冊（`tools.ts`）裡加一行
3. 完成

不需要改核心引擎、不需要改權限系統、不需要改 UI。因為所有系統都是對「合約」寫程式，不是對「特定工具」寫程式。

**商業洞察：** 當系統可以在不碰核心的情況下擴充功能，開發速度會隨時間加速。反過來，如果每加一個功能都要改十個地方，系統最終會慢到動不了。這是很多 SaaS 產品從快速迭代走向「每個 sprint 只能做一件小事」的根本原因。

### Hook 系統：不改引擎也能改行為

Claude Code 的 Hook 系統把「宣告而非控制」推到了使用者層面。

使用者可以在不改任何核心程式碼的情況下，注入自己的行為。想要在每次 AI 執行 `git push` 前跑一個驗證腳本？不需要改原始碼，在設定檔加一段：

```json
{
  "hooks": [{
    "event": "PreToolUse",
    "if": "Bash(git push)",
    "command": "validate-branch.sh"
  }]
}
```

而且 Hook 有多種來源，優先順序清楚：使用者設定 → 專案設定 → 企業政策 → 插件 → 內建預設。

**這是「開放封閉原則」——對擴充開放，對修改封閉。** 如果每次使用者想客製化行為都要改核心，會發生兩件事：核心越來越臃腫（要處理越來越多的特殊情況），使用者的客製化在每次升級時被覆蓋。Hook 讓核心保持穩定，讓客製化活在核心之外。

### Feature Flag：功能的開關也是宣告

`tools.ts`（工具清冊）裡，很多工具被包在條件判斷裡：

```
const SleepTool = feature('PROACTIVE') || feature('KAIROS')
  ? require('./tools/SleepTool/SleepTool.js').SleepTool
  : null
```

`feature('KAIROS')` 是一個遠端開關。如果沒開，`SleepTool` 等於 `null`。

這也是宣告式的設計：功能**宣告自己的存在條件**，系統根據這些宣告決定哪些功能活著。功能不控制自己什麼時候被啟用——決定權在系統（以及遠端的 feature flag 服務）手上。

## 支柱四：只為用到的東西付費

> 為「萬一某天需要」買單，是最昂貴的保險。

### Feature Flag 的另一面：死碼消除

上一節提到的 feature flag 不只是「功能開關」。更重要的是**死碼消除**（Dead Code Elimination）。

當建構工具看到 `feature('KAIROS')` 回傳 `false`，它不只是「不載入」——它會把整段 `require('./tools/SleepTool/...')` 從最終輸出中**物理移除**。使用者下載的版本裡，這段程式碼根本不存在。

目前至少有 12 個這樣的條件門檻：

- `COORDINATOR_MODE` — 多 agent 協作
- `KAIROS` — 常駐助手模式
- `AGENT_TRIGGERS` — 排程觸發
- `PROACTIVE` — 主動行為
- `MONITOR_TOOL` — 監控工具
- `KAIROS_GITHUB_WEBHOOKS` — GitHub 事件訂閱
- 還有 `BASH_CLASSIFIER`、`TEAMMEM`、`UNATTENDED_RETRY` 等

不是「隱藏起來但還佔空間」，而是**物理上不存在**。你不該為了還沒發生的事情付出代價。

**這跟精實創業的思維一致：不要為尚未驗證的假設分配資源。**

### 延遲載入：用到才搬，不用就別動

`main.tsx` 在啟動時，大型功能模組的載入全是條件式的：

```
const coordinatorModeModule = feature('COORDINATOR_MODE')
  ? require('./coordinator/coordinatorMode.js') : null
const assistantModule = feature('KAIROS')
  ? require('./assistant/index.js') : null
```

如果你不需要多 agent 協作，`coordinatorMode` 根本不會被載入。如果你沒開常駐助手，`assistant` 模組也不存在。

這和前面講的「循環依賴打斷」用的是同一個機制——延遲載入——但目的不同。打斷循環是為了**結構正確性**，這裡是為了**效能**。一個工具，兩個用途。

### 啟動優化：同時做不相依的事

`main.tsx` 啟動時，刻意把不相依的初始化步驟平行化：

- 讀取裝置管理設定
- 從金鑰圈預載憑證
- 初始化 feature flag 引擎
- 讀取使用者設定

這四件事彼此不需要等對方的結果，所以同時進行。原始碼的註解提到省下約 135 毫秒。

135 毫秒聽起來很少？但累積起來就是使用者體驗的差異。如果你每天開 20 次 Claude Code，一年省下的等待時間大約 16 分鐘。乘以全球使用者數，這是一個很實際的產品決策。

**核心哲學：不是「把所有功能都載進來然後隱藏沒用的」，而是在物理層面只載入需要的。** 跟雲端計算的按需付費是同一個思路——你不該為「萬一某天需要」持續買單。

## 四根柱子怎麼互相支撐

退一步看，這四個支柱不是獨立的——它們形成一個自洽的體系。

**預期失敗**告訴你心態：壞事會發生，要提前想好對策。
**隔離影響**告訴你結構：壞事發生時，不要讓它擴散。
**宣告而非控制**告訴你擴展方式：讓元件描述自己，系統根據描述做決定。
**只為用到的東西付費**告訴你資源觀：不存在的東西不會壞，也不會拖慢你。

它們之間有一條暗線：**都在最小化「一個改動影響其他地方」的機率。**

- 失敗被隔離，不擴散
- 依賴被切斷，不拖累
- 合約是穩定的，實作可以變
- 沒用的功能不存在，不出錯

這就是為什麼 1,900 個檔案的系統可以持續迭代，而不是每改一個地方就崩三個地方。

---

## 一句話總結

Anthropic 的工程師不是在追求完美——他們在系統化地**管理不完美**。預期失敗、隔離影響、宣告而非控制、只為用到的東西付費。

這四個原則不只適用於軟體，適用於任何你需要設計複雜系統的地方。
